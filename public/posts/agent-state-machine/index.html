Modeling an Agent Loop as a UML Behavioral State Machine

April 25, 2026 8-minute read
Technology
Agents • Uml • State-Machines • Llm

Why a Behavioral State Machine? Link to heading

Most agent loops in the wild are written as ad-hoc while loops with a pile of boolean flags: is_streaming, tool_pending, awaiting_user, cancelled. The flags drift, the edge cases multiply, and “what should happen if a cancel arrives mid-tool-call?” becomes a code-review argument.

A UML behavioral state machine removes the ambiguity. UML 2 gives us a precise vocabulary — simple state, composite state, pseudostate, transition, trigger, guard, effect, entry behavior, internal transition, extended state — and a well-defined run-to-completion semantics for processing events. Each rule lives in exactly one place, and the diagram is the specification.

This post models a typical LLM agent — text input, streaming output, tool calls, tool execution, human-in-the-loop confirmation, cancellation, and final outcomes — using UML conventions throughout. The runnable form is W3C SCXML, which is essentially UML state machines serialized as XML.

The State Model Link to heading

Pseudostates and Final States Link to heading

The diagram has one initial pseudostate (the filled circle) that designates the default substate of the top-level region. Three final states — terminated, failed, cancelled — represent terminal completions of the state machine, each carrying its own outcome semantics.

Composite State Link to heading

The state active is a composite state: it contains a region with five simple substates. Behavior shared by every substate (here, the response to a cancel trigger) is hoisted onto the composite, exploiting UML’s transition-priority rule that an outer transition fires when no inner transition is enabled. This is the UML equivalent of “handle it once, inherit everywhere.”

Simple States Link to heading

State Role Entry Behavior
idle ready —
awaiting_model active llm.complete
executing_tool active tool.invoke
awaiting_human waiting hil.ask
done settled —

Each entry behavior is invoked exactly once per entry, per UML semantics. Exit behaviors are symmetric (omitted here for brevity).

Extended State Link to heading

UML separates the state of the machine (which node is active) from its extended state (typed variables that survive across transitions). Four extended-state variables back this machine:

pending_tool_calls : Sequence<ToolCall>   // tool calls the model just emitted
pending_text       : String               // last user/assistant/error text
messages           : Sequence<Message>    // append-only conversation log
streaming_chunk    : String               // most recent streamed delta

Transitions read and update extended state via their effects.

The diagram visualizes self-loops for clarity, but two of them — idle → idle on session.restore and awaiting_model → awaiting_model on model.text_delta — are properly internal transitions in UML terms: they fire their effect without triggering exit/entry behaviors. This matters because awaiting_model’s entry behavior llm.complete must not run again on every streamed chunk.

The Run-to-Completion Step Link to heading

UML state machines process one event at a time using a run-to-completion (RTC) step:

  1. Dispatch the next event from the event pool.
  2. Find the innermost transition whose trigger matches and whose guard is true.
  3. Execute exit behaviors of states being left, the transition’s effect, then entry behaviors of states being entered.
  4. Settle in the new active configuration before dispatching the next event.

This is why concurrent cancel and tool.result cannot interleave halfway through a transition’s effect — RTC keeps the machine atomically consistent without locks.

Walking the Happy Path Link to heading

A turn with one tool call traces a deterministic path through the diagram:

  1. idle –[user.message / append user msg]–> awaiting_model. Exit behavior of idle runs; entry behavior of awaiting_model runs llm.complete.
  2. The model streams model.text_delta events. Each is an internal transition: streaming_chunk is updated and the hook.stream effect fires, but the active state and its entry behavior are untouched.
  3. The model emits model.tool_use. The transition to executing_tool records calls into pending_tool_calls and messages (effect), then enters executing_tool and runs tool.invoke (entry).
  4. tool.result transitions back to awaiting_model, appending outputs and clearing pending_tool_calls. Re-entry runs llm.complete so the model can continue with the new context.
  5. The model emits model.final → transitions to done, with the assistant message saved.
  6. done waits in the active configuration. A new user.message reopens the loop; session.finish produces a completion transition to the final state terminated.

Notable Modeling Choices Link to heading

These are the places UML’s vocabulary actually pays off:

  • Hoisted transition. cancel is declared on the composite state active. Per UML transition-selection rules, it fires whenever an inner transition is not enabled, so it works as a uniform interrupt over every substate without duplication.
  • Guarded transitions. session.restore is one trigger with five outgoing transitions guarded by [target = 'done'], [target = 'awaiting_human'], [target = 'failed'], [target = 'cancelled'], and the unguarded fallback. This is the standard UML pattern for routing on payload.
  • Internal vs. external transitions. model.text_delta and the unguarded session.restore are internal (no exit/entry rerun). model.final and model.tool_use are external (full exit/effect/entry sequence). The distinction is critical for entry behaviors with side effects like llm.complete.
  • Entry behaviors as side-effecting work. llm.complete, tool.invoke, and hil.ask are entry behaviors, not transition effects, so they’re invariant under whichever transition led here. Re-entering awaiting_model from executing_tool triggers a fresh llm.complete automatically.
  • Append-only extended state. Every effect that adds to messages does so by concat, making the conversation log monotonic. pending_tool_calls is cleared on every exit from executing_tool, preserving the invariant that no stale call IDs leak across turns.
  • Distinct final states. terminated, failed, and cancelled are separate UML final states rather than a single “ended” state with a status flag. The state machine’s outcome is part of its type.

The SCXML Realization Link to heading

W3C SCXML is the executable form of a UML behavioral state machine. The mapping is direct: <state> ↔ simple/composite state, <final> ↔ final state, <initial> / initial="…" ↔ initial pseudostate, <transition event= cond= target=> ↔ trigger/guard/target, <onentry> / <onexit> ↔ entry/exit behaviors, <datamodel> ↔ extended state.

<?xml version="1.0" encoding="UTF-8"?>
<scxml xmlns="http://www.w3.org/2005/07/scxml" version="1.0" datamodel="ecmascript" initial="active">
  <datamodel>
    <data id="pending_tool_calls" expr="[]"/>
    <data id="pending_text"       expr="''"/>
    <data id="messages"           expr="[]"/>
    <data id="streaming_chunk"    expr="''"/>
  </datamodel>

  <state id="active" initial="idle">
    <onentry><hook.enter state="active"/></onentry>
    <onexit><hook.exit state="active"/></onexit>
    <transition event="cancel" target="cancelled">
      <hook.transition event="cancel"/>
    </transition>

    <state id="idle">
      <onentry><hook.enter state="idle"/></onentry>
      <onexit><hook.exit state="idle"/></onexit>
      <transition event="session.restore" cond="_event.data.target == 'done'" target="done">
        <hook.transition event="session.restore"/>
        <assign location="messages" expr="_event.data.messages != null ? _event.data.messages : messages"/>
      </transition>
      <transition event="session.restore" cond="_event.data.target == 'awaiting_human'" target="awaiting_human">
        <hook.transition event="session.restore"/>
        <assign location="messages" expr="_event.data.messages != null ? _event.data.messages : messages"/>
      </transition>
      <transition event="session.restore" cond="_event.data.target == 'failed'" target="failed">
        <hook.transition event="session.restore"/>
        <assign location="messages" expr="_event.data.messages != null ? _event.data.messages : messages"/>
      </transition>
      <transition event="session.restore" cond="_event.data.target == 'cancelled'" target="cancelled">
        <hook.transition event="session.restore"/>
        <assign location="messages" expr="_event.data.messages != null ? _event.data.messages : messages"/>
      </transition>
      <transition event="session.restore">
        <hook.transition event="session.restore"/>
        <assign location="messages" expr="_event.data.messages != null ? _event.data.messages : messages"/>
      </transition>
      <transition event="user.message" target="awaiting_model">
        <hook.transition event="user.message"/>
        <assign location="pending_text" expr="_event.data.text"/>
        <assign location="messages" expr="messages.concat([{role: 'user', text: pending_text, ts: new Date().toISOString()}])"/>
      </transition>
    </state>

    <state id="awaiting_model">
      <onentry>
        <hook.enter state="awaiting_model"/>
        <llm.complete/>
      </onentry>
      <onexit><hook.exit state="awaiting_model"/></onexit>
      <transition event="model.text_delta">
        <assign location="streaming_chunk" expr="_event.data.text"/>
        <hook.stream/>
        <hook.transition event="model.text_delta"/>
      </transition>
      <transition event="model.final" target="done">
        <hook.transition event="model.final"/>
        <assign location="pending_text" expr="_event.data.text"/>
        <assign location="messages" expr="messages.concat([{role: 'assistant', text: pending_text, ts: new Date().toISOString()}])"/>
      </transition>
      <transition event="model.tool_use" target="executing_tool">
        <hook.transition event="model.tool_use"/>
        <assign location="pending_tool_calls" expr="_event.data.tool_calls != null ? _event.data.tool_calls : [{name: _event.data.name, args: _event.data.args}]"/>
        <assign location="messages" expr="messages.concat(pending_tool_calls.map(function(c) { return {role: 'tool_call', text: c.name, args: c.args, ts: new Date().toISOString()}; }))"/>
      </transition>
      <transition event="model.error" target="failed">
        <hook.transition event="model.error"/>
        <assign location="pending_text" expr="_event.data.message"/>
        <assign location="messages" expr="messages.concat([{role: 'error', text: pending_text, ts: new Date().toISOString()}])"/>
      </transition>
    </state>

    <state id="executing_tool">
      <onentry>
        <hook.enter state="executing_tool"/>
        <tool.invoke/>
      </onentry>
      <onexit><hook.exit state="executing_tool"/></onexit>
      <transition event="tool.result" target="awaiting_model">
        <hook.transition event="tool.result"/>
        <assign location="messages" expr="messages.concat(_event.data.outputs.map(function(o) { return {role: 'tool', text: o.output, ts: new Date().toISOString()}; }))"/>
        <assign location="pending_tool_calls" expr="[]"/>
      </transition>
      <transition event="tool.error" target="awaiting_model">
        <hook.transition event="tool.error"/>
        <assign location="pending_text" expr="_event.data.message"/>
        <assign location="messages" expr="messages.concat([{role: 'error', text: pending_text, ts: new Date().toISOString()}])"/>
        <assign location="pending_tool_calls" expr="[]"/>
      </transition>
      <transition event="hil.request" target="awaiting_human">
        <hook.transition event="hil.request"/>
      </transition>
    </state>

    <state id="awaiting_human">
      <onentry>
        <hook.enter state="awaiting_human"/>
        <hil.ask/>
      </onentry>
      <onexit><hook.exit state="awaiting_human"/></onexit>
      <transition event="user.response" target="executing_tool">
        <hook.transition event="user.response"/>
      </transition>
    </state>

    <state id="done">
      <onentry><hook.enter state="done"/></onentry>
      <onexit><hook.exit state="done"/></onexit>
      <transition event="user.message" target="awaiting_model">
        <hook.transition event="user.message"/>
        <assign location="pending_text" expr="_event.data.text"/>
        <assign location="messages" expr="messages.concat([{role: 'user', text: pending_text, ts: new Date().toISOString()}])"/>
      </transition>
      <transition event="session.finish" target="terminated">
        <hook.transition event="session.finish"/>
      </transition>
    </state>
  </state>

  <final id="terminated">
    <onentry><hook.enter state="terminated"/></onentry>
    <donedata>
      <param name="output" expr="pending_text"/>
    </donedata>
  </final>
  <final id="failed">
    <onentry><hook.enter state="failed"/></onentry>
  </final>
  <final id="cancelled">
    <onentry><hook.enter state="cancelled"/></onentry>
  </final>
</scxml>

Closing Thoughts Link to heading

Once the agent loop is a UML behavioral state machine, several recurring design questions answer themselves:

  • May the agent accept a new user message while a tool is running? — No. There is no transition out of executing_tool triggered by user.message; the event is deferred or silently consumed, depending on policy. Nothing in the contract is implicit.
  • Can the user cancel during human approval? — Yes. The cancel transition on the composite state active covers every substate, including awaiting_human.
  • What is the contract for resuming a saved session? — Exactly the five guarded session.restore transitions; their guards exhaustively partition the payload’s target values, and the unguarded fallback handles the rest.

The diagram is the specification, the SCXML is the implementation, and UML’s vocabulary is what lets us talk about either one without ambiguity.
